## 一、产品形态

### 1.1 整体结构

- **全屏滑动卡片**，每个维度独立一组 Slides，固定 6 张
- **移动端优先**，viewport 锁定不可缩放：`maximum-scale=1.0, user-scalable=no`
- **手势 & 键盘导航**：左右滑动 / 左右方向键 / 空格键
- **点击区域**：左半屏 = 上一张，右半屏 = 下一张（不可见的 nav 层覆盖全屏）

### 1.2 Slide 顺序（固定）

| 序号 | 页面名称 | 核心内容 |
|------|---------|---------|
| 1 | 开场 | 维度标题、钩子文案、来源 Chip |
| 2 | 古籍理论 | 经典引文 + 现代解读 |
| 3 | 来源 1 扫描 | 命盘逐字扫描动画 |
| 4 | 来源 2 分析 | 五行/互补数据可视化 |
| 5 | 分数 | 堆叠进度条 + 大分数 + Badge |
| 6 | 断语 | 结论卡片 |

### 1.3 顶部进度条

- 6 段等宽细条（高 3px，圆角 2px），底色 `rgba(255,255,255,0.12)`，填充色 `#fff`
- 每张 Slide 有独立停留时长（`DURS` 数组，单位 ms）
- 已读 Slide 进度条 100%，当前 Slide 实时动画填充，未读为 0%

## 三、通用动画系统

### 3.1 入场动画类

| 类名 | 动画效果 | 时长 | 用途 |
|------|---------|------|------|
| `.afu` | 淡入 + 上移 18px | 0.55s ease | 文字块、卡片内容 |
| `.afi` | 纯淡入 | 0.4s ease | 图片、分割线 |
| `.asi` | 淡入 + 缩放弹出（0.75→1） | 0.5s cubic-bezier(.34,1.56,.64,1) | 分数数字、Badge |

- 所有入场动画初始状态 `opacity:0`
- 延迟通过内联 `style="animation-delay:Xs"` 设置，逐元素错开 0.1～0.3s
- **重要**：Slide 切换时必须重置动画（见 3.3）

### 3.2 延迟规则

- 开场 Slide：0.2s / 0.4s / 0.8s / 1.1s / 1.4s 依次入场
- 扫描/分析 Slide：主内容 0.1～0.4s；数据驱动的动态元素（计数器、柱子点亮）由 JS setTimeout 控制，不走 CSS delay
- **分数页 Badge + 补充小字**：必须在数字滚动结束后出现
  - countUp 开始：2100ms；时长：1000ms；结束：3100ms
  - Badge `animation-delay` ≥ **3200ms**（3.2s）
  - 补充小字 `animation-delay` ≥ **3500ms**（3.5s）

### 3.3 动画重置机制（关键技术约束）

每次 Slide 激活时，必须重置所有 `.afu/.afi/.asi` 元素的动画，使其从头播放。  
**已知坑**：`el.style.animation = 'none'` 是 CSS 简写属性，会将 `animation-delay` 子属性重置为 0s，覆盖内联设置的延迟。

**正确写法**：
```js
card.querySelectorAll('.afu,.afi,.asi').forEach(el => {
  const d = el.style.animationDelay;   // 先保存延迟
  el.style.animation = 'none';
  el.offsetHeight;                      // 强制 reflow
  el.style.animation = '';
  if (d) el.style.animationDelay = d;  // 还原延迟
});
```

### 4.5 Slide 5 · 分数

**布局**（从上到下）：

```

    ↓

    ↓
```

**堆叠进度条**：

- 胶囊，圆角 100px，Syne 700 19px，绿色系底色+边框

- 用 JS 在 `activate()` 时通过保存/还原机制保持 delay（见 3.3）

**补充小字**：
- 12px，`rgba(255,255,255,.35)`，居中，max-width 280px
- `animation-delay: 3.5s`，`.afu` 类，同样受保存/还原机制保护

## 五、文案规范

### 5.1 通用原则

- **写「在一起会发生什么」，不写「他们是什么关系」**
- 短句 > 长句，具体细节 > 抽象描述
- 高分夸张（让人觉得运气好到不像话），低分要毒（说出不敢承认但心知肚明的真相），中间段平实
- 禁止使用 AI 填充词：「此外」「值得注意的是」「不仅…而且」「体现了」「彰显了」
- 禁止公式化结尾（「未来充满希望」「这段关系潜力无限」）

### 5.2 维度标签禁令

- **严禁在任何可见 UI 中出现「L1」「L2」「L3」「L4」「L5」等工程编号**
- 来源标签用中文描述：「印星占比」「五神互补」「日干对位」等

### 5.3 Badge 文案结构

每个 Badge 由两部分组成：
1. **Badge 标签**（胶囊内大字）：2～6 字，直接定性
2. **补充小字**（Badge 下方）：一句话，具体场景化，带语气

## 六、评分与数据规范

### 6.1 得分区间

所有维度统一：**-100% ～ +100%**，向右（正方向）为好。

### 6.2 堆叠条加权规则

进度条显示的是**加权贡献值，不是原始分**。

```
总分 = 来源1 × 权重A + 来源2 × 权重B
S1 宽度% = 来源1原始分 × 权重A
S3 宽度% = 来源2原始分 × 权重B
S1 + S3 = 总分（即两段之和等于仪表盘显示的大数字）
```

> 这样两段宽度之和直接 = 总百分比，进度条不会溢出，含义直观。

### 6.3 各维度来源权重

| 维度 | 来源 1 | 权重 | 来源 2 | 权重 |
|------|--------|------|--------|------|
| DIM1 同频指数 | 日干对位 (L1) | 55% | 全局十神 (L3 同频字) | 45% |
| DIM2 义气指数 | 印星占比 (L3 印枭) | 55% | 五神互补 (L4) | 45% |
| DIM3 消耗指数 | 劫煞占比 (L3 劫/杀/比肩) | 50% | 忌神激活 (L4 负向) | 50% |
| DIM4 成长指数 | 调候互补 (L4 正向) | 60% | 贵人神煞 (L5) | 40% |
| DIM5 缘分指数 | 月支社交宫对位 (L5-A) | 55% | 神煞互见 (L5-B 天乙/桃花/文昌) | 45% |

### 6.4 Badge 得分可达性分析

> 各维度计算来源决定了实际可达的分数区间，Badge 分档必须与此对齐，否则存在"永远不可触发"的死档。

#### DIM1 同频指数

| 来源 | 实际范围 |
|------|---------|
| L1 日主对位 | [−35, +100]（才/杀最差 −35，比/比+五合+化 最好 +100） |
| L3 同频十神（比/食/伤 only，只计正向） | [0, +100] |
| **DIM1 合计** | **[−35, +100]** |

Badge 可达性：
- `命定知己 ✦` ≥+80 ✅ 可达
- `同频搭子 ✦` +50~+79 ✅ 可达
- `能聊，有限` +20~+49 ✅ 可达
- `频道不太对` −19~+19 ✅ 可达
- `频道对不上` <−20 ⚠️ 理论可达（L1=−35, L3=0 → −35），**实践中极罕见**（L1 极负时对方八字里往往有散布的比/食/伤，L3 很少真到 0）。保留此档但属于尾部极值。

#### DIM2 义气指数

| 来源 | 实际范围 |
|------|---------|
| L3 印星部分（印/枭 only，只计正向） | [0, +100] |
| L4 五神互补全量（可正可负） | [−100, +100] |
| **DIM2 合计** | **[−100, +100]** |

Badge 全部可达 ✅。高档「命中真贵人 ✦」需 L3 印星高 × L4 双向正向同时成立，概率低但不为零。

#### DIM3 消耗指数 ⚠️ 结构性修正

两个来源均为「负向专项」：

| 来源 | 实际范围 |
|------|---------|
| L3 劫煞占比（劫/杀/比肩，只计负向） | [−100, **0**] |
| L4 忌神激活（忌/仇触发，只计负向） | [−100, **0**] |
| **DIM3 实际上限** | **0**（最好情况 = 彼此无消耗） |

**结论：DIM3 正向区间（>0）在数学上不可达。** 之前规格包含"轻度充电"、"天然充电宝"等正向 Badge，已删除。Badge 档位已修订为纯负向 5 档（见下表）。充电/成长语义由 DIM4 承载。

**修订后 DIM3 Badge 分档（[−100, 0] 区间）：**

| 分数段 | Badge |
|--------|-------|
| 0 ～ −12 | 基本不耗 |
| −13 ～ −38 | 轻度消耗 |
| −39 ～ −65 | 慢慢耗着 |
| −66 ～ −85 | 能量黑洞 |
| ≤ −86 | 相互摧毁 |

> Badge 小字完整文案见 `bazi-compatibility-scoring-spec.md` § DIM3 模块 B。

> Demo l3-demo.html 使用 `dim3Score: −35`，对应修订后「轻度消耗」档（−13～−38）。

## 七、JS 架构约束

### 7.1 文本引擎

每个维度一个 `getDIMxTexts({...})` 函数，接收计算好的参数，返回：
```js
{ scanCaption, badge, badgeSubtext, verdictTitle, verdictBody[] }
```

`renderTexts()` 在页面加载时调用一次，将所有文本填入 DOM。

### 7.2 卡片 Setup 函数

每次激活对应 Slide 时调用：
- `setupScan()` → Slide 3
- `setupComp()` / `setupL1()` → Slide 4
- `setupScore()` → Slide 5

**注意**：每次重新激活 Slide 时，JS 的 setTimeout 会重新堆积，需在 setup 开头重置所有 DOM 状态（进度条宽度归 0、计数器 opacity 归 0、`.lit`/`.visible` 类清除等）。

### 7.3 导航

```js
function prev(){ if(cur>0){ cur--; activate(cur); } }
function next(){ if(cur<TOTAL-1){ cur++; activate(cur); } }
```

事件：`touchstart`/`touchend`（滑动 > 40px 触发）+ `keydown`（←/→/空格）。
